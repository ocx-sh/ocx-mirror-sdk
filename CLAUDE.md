# CLAUDE.md

Guide for Claude Code when working in this repo.

## What this is

`ocx-mirror-sdk` is the Python SDK for authoring [`ocx-mirror`](https://github.com/ocx-sh/ocx) generator scripts. `ocx-mirror` (the Rust binary in the main `ocx` repo) ingests upstream tool releases and republishes them as OCI artifacts. For sources `ocx-mirror` cannot crawl directly, a small Python generator emits a `url_index` JSON document; this SDK provides the typed building blocks for those generators.

The entire repository is the SDK — there are no subsystems. See `.claude/rules/architecture.md` for the module map.

## Public API

| Symbol | Purpose |
|---|---|
| `IndexBuilder` | Typed builder for the `url_index` JSON manifest |
| `list_releases` | Iterate GitHub releases via REST (`github3.py`) |
| `list_releases_graphql` | Iterate GitHub releases via GraphQL (use on large repos that 504 on REST) |
| `Asset`, `Release` | Typed views of GitHub release assets |
| `extract_urls` | Pull download URLs from release notes |
| `FileCache` | Disk-backed HTTP response cache, 1h default TTL |

## Relationship to `ocx-sh/ocx`

The `url_index` JSON format is **owned by** the `ocx_mirror` Rust crate in the main `ocx` repo (`crates/ocx_mirror/src/source/url_index.rs::RemoteIndex`). Its JSON Schema is generated by `cargo run -p ocx_mirror --features jsonschema -- schema url-index` and published at <https://ocx.sh/schemas/url-index/v1.json>.

This SDK is a **downstream consumer**. The schema flow:

1. Schema author updates the Rust type → regenerates → website publishes the new JSON.
2. SDK maintainer runs `ocx run -- task codegen` here → `_schema.py` regenerates from the URL.
3. Bump SDK version, cut a release. Mirror repos update their pins.

When the schema gets a breaking change (`/v2.json`), this SDK ships a new major version.

## Toolchain bootstrap (OCX dogfood)

This repo manages its own dev toolchain through OCX. Install OCX once:

```bash
curl -sSL https://setup.ocx.sh | sh
```

`ocx.toml` declares `task` and `uv`. Run every command through `ocx run`:

```bash
ocx run -- task verify    # format check + lint + types + tests
ocx run -- task test
ocx run -- task codegen   # regenerate _schema.py
ocx run -- task format    # apply ruff formatter
```

Python linters (`ruff`, `pyright`) live in `[project.optional-dependencies] dev` in `pyproject.toml` — `uv` pulls them at sync time.

Optional shell convenience: `eval "$(ocx env --shell=sh)"` once per session, then drop the `ocx run --` prefix.

## Build & dev commands

| Task | Purpose |
|---|---|
| `task verify` | Full quality gate — run before commit |
| `task test` | pytest only |
| `task lint` | ruff check |
| `task types` | pyright |
| `task format` | apply ruff formatter |
| `task format:check` | check formatting (used by verify) |
| `task codegen` | Re-fetch JSON Schema + regenerate `_schema.py` |
| `task schema:fetch` | Just re-fetch the JSON Schema |

## Distribution

Consumers pin via PEP 723 inline metadata or `[tool.uv.sources]` git dep — see `README.md`. PyPI publish is deferred until post-v1.0.

## Stability

Pre-1.0. Breaking changes ship without migration shims. No deprecation warnings, no compatibility layers.

## Core principles

1. **Understand first** — read before write, grep before create.
2. **Prove it works** — failing test before fix; `task verify` green before commit.
3. **Keep it safe** — no secrets in code; validate external input at boundaries.
4. **Keep it simple** — small functions, single responsibility, delete dead code.
5. **Don't repeat yourself** — extract only when duplication is real, not incidental.
6. **Ship it** — work on branches, never push to main without review.
7. **Leave a trail** — name things so the next person understands.
8. **Learn and adapt** — turn recurring feedback into rule updates.
